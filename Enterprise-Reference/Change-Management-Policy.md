\# Change Management Policy



\---



\# Apex Manufacturing, Inc.



\## Purpose



This document establishes the Change Management Policy for Apex Manufacturing, Inc.



The purpose of Change Management is to ensure that all changes to the organization's technology environment are planned, tested, approved, documented, implemented, and reviewed in a controlled manner. Following a standardized process minimizes service disruptions, reduces risk, and maintains system stability.



This policy applies to all production systems, infrastructure, applications, network devices, cloud resources, and security technologies managed by the Information Technology department.



\---



\# Scope



This policy applies to changes involving:



\- Servers

\- Active Directory

\- Microsoft 365

\- Azure resources

\- Network infrastructure

\- Firewalls

\- VPN services

\- Endpoint management

\- Enterprise applications

\- Help Desk systems

\- Security tools

\- Backup systems



\---



\# Change Objectives



The Change Management process aims to:



\- Reduce operational risk

\- Minimize downtime

\- Improve service reliability

\- Maintain accurate documentation

\- Ensure accountability

\- Support business continuity

\- Provide rollback capability

\- Improve communication across IT teams



\---



\# Change Types



\## Standard Change



Low-risk, pre-approved changes that follow established procedures.



Examples:



\- User account creation

\- Password resets

\- Group membership updates

\- Printer deployments

\- Approved software installations



Approval Required:



\- Help Desk Supervisor or Systems Administrator



\---



\## Normal Change



Changes that require planning, testing, approval, and scheduling.



Examples:



\- Server configuration changes

\- Group Policy modifications

\- Network configuration updates

\- New application deployments

\- Storage expansion



Approval Required:



\- IT Infrastructure Manager



\---



\## Emergency Change



Changes necessary to restore critical business services or address active security threats.



Examples:



\- Critical security patch deployment

\- Server outage recovery

\- Firewall rule updates during an incident

\- Malware containment

\- Account compromise response



Approval Required:



\- IT Infrastructure Manager or Chief Information Officer



Emergency changes must be documented and reviewed after implementation.



\---



\# Change Management Process



All normal changes should follow the process below.



```text

Request



↓



Risk Assessment



↓



Technical Review



↓



Approval



↓



Testing



↓



Implementation



↓



Validation



↓



Documentation



↓



Change Closure

```



\---



\# Change Request Requirements



Every change request should include:



\- Change title

\- Business justification

\- Systems affected

\- Planned implementation date

\- Estimated downtime

\- Risk assessment

\- Rollback plan

\- Testing procedure

\- Validation steps

\- Requestor

\- Approver



\---



\# Risk Classification



| Risk Level | Description |

|------------|-------------|

| Low | Minimal impact with documented procedures |

| Medium | Moderate business impact requiring planning |

| High | Significant impact requiring executive approval |



\---



\# Maintenance Windows



Scheduled maintenance should occur during the approved maintenance window.



| Day | Time |

|-----|------|

| Saturday | 10:00 PM – 2:00 AM Central Time |



Emergency maintenance may occur outside the standard maintenance window when business operations require immediate action.



\---



\# Testing Requirements



Before implementation, changes should be tested whenever practical.



Testing should verify:



\- Expected functionality

\- Compatibility

\- Security controls

\- Performance

\- User access

\- Backup integrity



\---



\# Rollback Requirements



Every normal change must include a documented rollback plan.



Rollback plans should identify:



\- Recovery steps

\- Estimated recovery time

\- Required personnel

\- Backup requirements

\- Validation procedures



Rollback procedures should be tested whenever feasible.



\---



\# Communication



For planned changes, stakeholders should be notified of:



\- Scheduled maintenance

\- Expected impact

\- Estimated downtime

\- Completion status

\- Unexpected issues



Communication methods may include:



\- Email

\- Microsoft Teams

\- Help Desk announcements

\- Internal IT notifications



\---



\# Post-Implementation Review



Following implementation, the responsible technician should verify:



\- Change completed successfully

\- Services are operating normally

\- Monitoring shows no issues

\- Documentation has been updated

\- Users have been notified (if applicable)



High-risk and emergency changes should include a formal post-implementation review.



\---



\# Documentation Requirements



The following records should be retained for each completed change:



\- Approved Change Request

\- Risk Assessment

\- Implementation Notes

\- Validation Results

\- Rollback Documentation (if used)

\- Post-Implementation Review



\---



\# Roles and Responsibilities



| Role | Responsibility |

|------|----------------|

| Help Desk Technician | Submit and document standard changes |

| Systems Administrator | Implement approved infrastructure changes |

| Network Administrator | Implement network-related changes |

| IT Infrastructure Manager | Review and approve normal changes |

| Chief Information Officer | Approve major or emergency changes and provide executive oversight |



\---



\# Policy Compliance



Failure to follow the Change Management process may result in:



\- Service disruptions

\- Security risks

\- Audit findings

\- Unauthorized configuration changes

\- Disciplinary action, when appropriate
